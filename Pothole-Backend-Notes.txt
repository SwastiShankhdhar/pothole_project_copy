## What are Migrations and Why Do We Need Them?

---

### The Core Concept

You wrote this in `detection/models.py`:

```python
class DetectionResult(models.Model):
    input_type      = models.CharField(...)
    input_image     = models.ImageField(...)
    pothole_count   = models.PositiveIntegerField(...)
    detections_json = models.JSONField(...)
    created_at      = models.DateTimeField(...)
```

This is **Python code**. But PostgreSQL doesn't understand Python — it only understands **SQL**. Someone needs to translate this Python class into a SQL table.

**Migrations are that translator.**

---

### What Actually Happens — Step by Step

```
Your models.py (Python)
        ↓
python manage.py makemigrations
        ↓
0001_initial.py is generated   ← translation file
        ↓
python manage.py migrate
        ↓
PostgreSQL executes this SQL:

CREATE TABLE detection_detectionresult (
    id              SERIAL PRIMARY KEY,
    input_type      VARCHAR(10),
    input_image     VARCHAR(100),
    pothole_count   INTEGER,
    detections_json JSONB,
    created_at      TIMESTAMP
);
```

You never write that SQL manually — Django generates and runs it for you.

---

### Why Not Just Write SQL Directly?

Because your models will **change over time**. For example next week you might add a `severity` field:

```python
class DetectionResult(models.Model):
    ...
    severity = models.CharField(max_length=20)  # NEW FIELD
```

Django tracks this change and generates:
```sql
ALTER TABLE detection_detectionresult 
ADD COLUMN severity VARCHAR(20);
```

That becomes `0002_add_severity.py`. Every change to your model gets its own numbered migration file — giving you a **complete history** of how your database evolved.

---

### How It Links to Your Project Specifically

```
detection/models.py          →  defines DetectionResult table
        ↓ makemigrations
detection/migrations/
    __init__.py              →  makes migrations a Python package
    0001_initial.py          →  contains the SQL translation
        ↓ migrate
PostgreSQL: pothole_db
    table: detection_detectionresult   ← actually created here
```

When your API receives an image and runs YOLO — this is what happens with the database:

```python
# in detection/views.py
record = DetectionResult(          # create a Python object
    pothole_count=3,
    detections_json=[...],
    ...
)
record.save()                      # Django translates this to:
                                   # INSERT INTO detection_detectionresult 
                                   # VALUES (3, '[...]', ...)
```

Without migrations, that table doesn't exist in PostgreSQL — so `record.save()` crashes.

---

### The Migration Files Folder

After `makemigrations` your detection app looks like this:

```
detection/
    migrations/
        __init__.py        ← empty, makes it a package
        0001_initial.py    ← auto-generated, DO NOT edit manually
    models.py
    views.py
    serializers.py
    ...
```

The `0001_initial.py` file should be **committed to Git** — it's part of your codebase, not a temp file. When your teammate clones the repo and runs `migrate`, they get the exact same database structure.

---

### Summary

| Command | What it does |
|---|---|
| `makemigrations` | Reads `models.py`, generates the SQL translation file |
| `migrate` | Executes that SQL against PostgreSQL, creates/updates tables |
| Migration files | Version history of your database structure |
| Without migrations | Tables don't exist → every `record.save()` crashes |

Both outputs are perfectly valid secret keys — they are **supposed to look like random gibberish**. That's the whole point. A secret key that looks readable would be a weak secret key.

Just copy either one and paste it directly into your `.env` file. I recommend the second one since it has the `django-insecure-` prefix which is Django's own convention:

```env
DEBUG=True
SECRET_KEY=django-insecure-b165345c58a31d9ecf1370397a144323d37fe388494b943ad8edbbcf515d17ef

DB_NAME=pothole_db
DB_USER=postgres
DB_PASSWORD=your_postgres_password_here
DB_HOST=localhost
DB_PORT=5432

MODEL_PATH=models/pothole.pt
MEDIA_ROOT=media/
```

You never need to read or remember it — Django reads it silently in the background every time the server starts. You as the developer never type it manually anywhere. It just sits in `.env` and does its job invisibly.

Think of it like a house key — you don't need to memorize its shape, you just keep it safe and use it to unlock the door.

**The only two rules:**
- Never share it publicly or push it to GitHub
- Never change it in production — it will invalidate all existing sessions and tokens
